# golang-server

A Go project scaffold following official Go project layout conventions and interface-based design for testability.

## Setup

```bash
go mod tidy
cp .env.example .env
```

After cloning, replace the module path `github.com/yourorg/yourproject` in:
- `go.mod`
- `.mockery.yaml`
- All import statements

## Commands

```bash
make fmt        # Format Go source files
make vet        # Check formatting and run go vet
make test       # Run all tests (runs vet first)
make build      # Build the server binary
make generate   # Regenerate mocks with mockery
make help       # Show all available targets
```

## Project Structure

Based on the [official Go project layout](https://go.dev/doc/modules/layout) and community standard [golang-standards/project-layout](https://github.com/golang-standards/project-layout):

```
golang-server/
├── cmd/
│   └── server/          # Application entry point (main.go)
├── internal/            # Private application code (not importable by other projects)
│   ├── domain/          # Core domain types and business entities
│   ├── models/          # Data models, DTOs, request/response types
│   ├── services/        # Business logic and orchestration layer
│   ├── clients/         # External API/database clients (behind interfaces)
│   ├── mocks/           # Generated mock implementations (via mockery)
│   └── utils/           # Shared helpers and utility functions
├── tests/               # Integration and end-to-end tests
├── .mockery.yaml        # Mockery configuration for mock generation
├── Makefile             # Build, test, and development commands
├── go.mod               # Module definition and dependencies
└── .github/workflows/   # CI pipeline
```

### Key layout principles (from official Go docs)

- **`cmd/`**: Each subdirectory contains a `main.go` for a single binary. Keep these minimal -- delegate to `internal/` packages.
- **`internal/`**: Code here cannot be imported by other modules. This is enforced by the Go compiler. Use this for all application-specific logic.
- **No `pkg/`**: The official Go blog [recommends against](https://go.dev/doc/modules/layout#package-or-command-with-supporting-packages) a top-level `pkg/` directory. If you need exportable libraries, place them at the module root or in descriptively named directories.
- **Flat when possible**: Don't create packages prematurely. Start with fewer packages and split when you have a clear reason (circular dependency, distinct responsibility).

Create only the packages you need -- not all projects require every layer.

## Interface-Based Design and Mocking

All external dependencies (HTTP clients, database connections, CLI commands) should be accessed through interfaces. This enables clean testing with mockery-generated mocks.

### Pattern

```go
// internal/clients/http.go

// 1. Define the interface
type HTTPClient interface {
    Do(req *http.Request) (*http.Response, error)
}

// 2. Production implementation
type realHTTPClient struct{ client *http.Client }

func (c *realHTTPClient) Do(req *http.Request) (*http.Response, error) {
    return c.client.Do(req)
}

// 3. Package-level variable (default to production impl)
var client HTTPClient = &realHTTPClient{client: http.DefaultClient}

// 4. Test swap function (returns restore function for cleanup)
func SetTestClient(c HTTPClient) func() {
    orig := client
    client = c
    return func() { client = orig }
}
```

### Using mocks in tests

```go
// internal/services/myservice_test.go

func TestMyService(t *testing.T) {
    m := mocks.NewMockHTTPClient(t)
    restore := clients.SetTestClient(m)
    defer restore()

    m.On("Do", mock.AnythingOfType("*http.Request")).Return(&http.Response{
        StatusCode: http.StatusOK,
        Body:       io.NopCloser(strings.NewReader(`{"key":"value"}`)),
    }, nil)

    // Call the function under test
    result, err := myservice.Fetch(context.Background(), "https://api.example.com/data")
    assert.NoError(t, err)
    assert.Equal(t, "value", result.Key)

    m.AssertExpectations(t)
}
```

### Adding new mocks

1. Define your interface in the appropriate `internal/` package
2. Add the package and interface to `.mockery.yaml`
3. Run `make generate`
4. Import the generated mock from `internal/mocks/` in your tests

## Testing Conventions

- **No subprocess spawning**: Tests must never create real external processes (git, curl, etc.)
- **Always use mocks**: Generated by mockery from `internal/mocks/`
- **CGO disabled**: Tests run with `CGO_ENABLED=0` to avoid macOS AMFI issues
- **Race detector**: For CI with race detection, use `CGO_ENABLED=1 go test -race ./...` (requires codesigned binaries on macOS)
- **Integration tests**: Tests in `tests/` should use `//go:build integration` and be run separately with `go test -tags integration ./tests/...`

## macOS (Darwin) Codesigning

On macOS, Go binaries built with the race detector (`-race`) must be codesigned to satisfy Apple's AMFI (Apple Mobile File Integrity) framework. Without signing, the race-detector-instrumented binary may be killed at runtime.

The `make build` target automatically codesigns the binary on Darwin:

```bash
codesign --sign - --force --preserve-metadata=entitlements bin/server
```

If you need to run tests with the race detector on macOS:

```bash
CGO_ENABLED=1 go test -race ./...
```

Ensure the test binary is codesigned, or use `CGO_ENABLED=0` (the default in `make test`) to avoid the issue entirely.
